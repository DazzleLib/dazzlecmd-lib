# Human Test Checklist: FQCN arc slices 1-2 — property store + bang-path grammar

**Target version:** dazzlecmd-lib `0.10.0-alpha` / `0.10.1-alpha` (branch `fqcn-unification`)
**Ships with commits:** `f07e9b5` (PropertyStore + ConfigManager filename/filekit), `707fb37` (fqcn_grammar parser)
**Companion automated tests:** `tests/test_property_store.py` (15), `tests/test_fqcn_grammar.py` (42); full lib suite **342 passing**.

## What this is

The FQCN unification arc's first two slices are **foundations** — they add a property store and a bang-path parser but do not yet change any user-facing CLI surface. The one real risk is that Slice 1 reroutes `ConfigManager`'s atomic write through `dazzle_filekit.atomic_write_json` — and `ConfigManager` is on the **hot path for every config read/write** (kit enable/disable/attach, favorites, visibility). This checklist's primary job is to confirm **the live editable-installed `dz` is unharmed** by that file-I/O change, plus exercise the new store/parser. Run by a maintainer before merging the branch.

## Prerequisites

1. Windows (primary) or POSIX; Python 3.12; `dazzlecmd-lib` installed editable on branch `fqcn-unification`; `dazzle-filekit>=0.2.4` installed (`python -c "import dazzle_filekit"` succeeds).
2. **State isolation is mandatory** — every step that writes config MUST point `DAZZLECMD_CONFIG` at a throwaway temp file. NEVER run the write steps against your real `~/.dazzlecmd/config.json` or `~/.dz/config.json`.
3. Reset between sections: delete the temp config/properties files (commands in each section).

> **Production-safety note:** the read-only smoke steps (HV.1) use your real install on purpose (to prove it still loads). Every *write* step uses a temp `DAZZLECMD_CONFIG`. Keep that line straight and your real dotfiles are never touched.

## High-Value Verification (~5 minutes)

### HV.1 — Live `dz` still loads + reads config (PRODUCTION SAFETY)

Validates: the filekit-backed `ConfigManager` didn't break the engine's config read on the hot path.

- [ ] **Steps** (same across all shells, `dz` only, read-only):
  ```
  dz --version
  dz list
  dz info
  ```
- [ ] **Expected:** `--version` prints `0.11.x`; `dz list` shows the tool list; `dz info` renders; all exit 0. No tracebacks, no "could not read config" warnings.
- [ ] **Why this matters:** if the filekit refactor broke config reads, every `dz` invocation that touches config would fail — this is the blast-radius smoke.

### HV.2 — Real config WRITE round-trips (PRODUCTION SAFETY, temp config)

Validates: `ConfigManager.write` → `atomic_write_json` produces a valid, re-readable config file.

- [ ] **Steps:**

  **cmd.exe**:
  ```cmd
  set DAZZLECMD_CONFIG=%TEMP%\fqcn_check.json
  if exist %DAZZLECMD_CONFIG% del %DAZZLECMD_CONFIG%
  dz kit disable media
  dz list
  type %DAZZLECMD_CONFIG%
  ```
  **PowerShell**:
  ```powershell
  $env:DAZZLECMD_CONFIG = "$env:TEMP\fqcn_check.json"
  if (Test-Path $env:DAZZLECMD_CONFIG) { Remove-Item $env:DAZZLECMD_CONFIG }
  dz kit disable media
  dz list
  Get-Content $env:DAZZLECMD_CONFIG
  ```
  **POSIX**:
  ```bash
  export DAZZLECMD_CONFIG=/tmp/fqcn_check.json
  rm -f "$DAZZLECMD_CONFIG"
  dz kit disable media
  dz list
  cat "$DAZZLECMD_CONFIG"
  ```
- [ ] **Expected:** the disable persists (re-running `dz list` reflects it), and the temp config is **valid JSON** with `indent=4` and a trailing newline (open it — it should be human-readable, not truncated/corrupt). A `.tmp` sibling should NOT be left behind.
- [ ] **Why this matters:** this is the exact code path the filekit refactor changed — a real subprocess writing a real file atomically.

### HV.3 — PropertyStore round-trips by bang-path

Validates: Slice 1's store get/set against a separate `properties.json`.

- [ ] **Steps** (any shell):
  ```
  python -c "from dazzlecmd_lib.property_store import PropertyStore; import tempfile,os; d=tempfile.mkdtemp(); s=PropertyStore(config_dir=d); s.set('dz:.kit.channels.verbosity',4); print('get:',s.get('dz:.kit.channels.verbosity')); print('file:', os.path.exists(os.path.join(d,'properties.json')))"
  ```
- [ ] **Expected:** `get: 4` and `file: True`.
- [ ] **Why this matters:** the store is the foundation everything later reads/writes.

### HV.4 — Property writes do NOT touch config.json

Validates: the separate-file decision (no cross-contamination).

- [ ] **Steps** (any shell):
  ```
  python -c "from dazzlecmd_lib.config import ConfigManager; from dazzlecmd_lib.property_store import PropertyStore; import tempfile; d=tempfile.mkdtemp(); c=ConfigManager(config_dir=d); c.write({'active_kits':['media']}); PropertyStore(config_dir=d).set('dz:.x',1); c.invalidate(); print('config active_kits:', c.read().get('active_kits')); print('dz:.x in config?', 'dz:.x' in c.read())"
  ```
- [ ] **Expected:** `config active_kits: ['media']` and `dz:.x in config? False`.
- [ ] **Why this matters:** confirms properties and operational config stay cleanly separated.

### HV.5 — Grammar parses the operators

Validates: Slice 2's parser over `:` / `.` / `:.` / `:+`.

- [ ] **Steps** (any shell):
  ```
  python -c "from dazzlecmd_lib.fqcn_grammar import parse, unparse; p=parse('dz:.kit.channels.verbosity'); print(p.root, [(s.op,s.name) for s in p.segments]); print('roundtrip:', unparse(p)=='dz:.kit.channels.verbosity')"
  ```
- [ ] **Expected:** `dz [(':.','kit'), ('.','channels'), ('.','verbosity')]` and `roundtrip: True`.
- [ ] **Why this matters:** the parser is the addressing foundation for every later slice.

## Section 1: PropertyStore (Slice 1)

### 1.1 delete is surgical
- [ ] **Steps:** `python -c "from dazzlecmd_lib.property_store import PropertyStore; import tempfile; s=PropertyStore(config_dir=tempfile.mkdtemp()); s.set('dz:.a',1); s.set('dz:.b',2); print('deleted:',s.delete('dz:.a')); print('a:',s.get('dz:.a'),'b:',s.get('dz:.b'))"`
- [ ] **Expected:** `deleted: True`, `a: None b: 2` (only `dz:.a` removed).

### 1.2 list_prefix returns a family, skips bookkeeping
- [ ] **Steps:** set `dz:.kit.channels.verbosity`, `dz:.kit.channels.config.verbosity`, `dz:.tool.channels.verbosity`; call `list_prefix('dz:.kit.channels')`.
- [ ] **Expected:** only the two `dz:.kit.channels*` keys; no `_schema_version`.

### 1.3 per-aggregator isolation
- [ ] **Steps:** two `PropertyStore`s with different `config_dir`s; set the same key to different values.
- [ ] **Expected:** each keeps its own value (wtf-style `~/.wtf/properties.json` would be independent of `~/.dz/properties.json`).

### Reset
- Delete temp dirs created above (they are under `%TEMP%` / `/tmp`).

## Section 2: ConfigManager back-compat + isolation (Slice 1)

### 2.1 default filename unchanged
- [ ] `ConfigManager(config_dir=d).config_path()` ends with `config.json` (existing callers see no change).

### 2.2 DAZZLECMD_CONFIG isolates BOTH files
- [ ] With `DAZZLECMD_CONFIG=<dir>/custom.json`: a default `ConfigManager()` resolves to `custom.json`; a `ConfigManager(filename="properties.json")` resolves to `<dir>/properties.json` (sibling, same temp dir) — one env var isolates both.

### 2.3 replace() deletes wholesale
- [ ] `cm.write({'a':1,'b':2})` then `cm.replace({'a':1})` then re-read → `b` is gone (whereas `write` would have merged it back).

## Section 3: Grammar parser (Slice 2)

### 3.1 compound-first tokenizing
- [ ] `parse('dz:.kit')` → first segment op is `:.` (NOT `:` + `.`); `parse('dz:+kit')` → op `:+`.

### 3.2 malformed rejected
- [ ] Each of these raises `FQCNParseError`: `''`, `':kit'`, `'dz:'`, `'dz::kit'`, `'dz:.KIT'` (uppercase), `'dz..x'`, `'dz:-bad'`.

### 3.3 round-trip
- [ ] `unparse(parse(x)) == x` for `dz:wtf:.kit`, `dz:.verb:supra:management.channels.verbosity`, `dz:grep.recipe-1`.

## Section 4: PRODUCTION SAFETY (the point of this checklist)

### 4.1 kit enable/disable round-trip against a TEMP config
- [ ] **Steps:** (use the HV.2 temp-config setup) `dz kit disable <a-kit>` → `dz list` (reflects it) → `dz kit enable <a-kit>` → `dz list` (restored). Inspect the temp config between steps.
- [ ] **Expected:** state persists and restores; the temp `config.json` stays valid JSON throughout; no `.tmp` leftovers; no warnings.

### 4.2 your REAL config is untouched
- [ ] **Steps:** after the temp-config tests, **unset** `DAZZLECMD_CONFIG` and run `dz list`.

  **cmd.exe**: `set DAZZLECMD_CONFIG=` · **PowerShell**: `Remove-Item Env:\DAZZLECMD_CONFIG` · **POSIX**: `unset DAZZLECMD_CONFIG`
- [ ] **Expected:** your normal tool list — your real config was never written during this checklist.

### 4.3 BOM / pre-existing config tolerance
- [ ] **Steps:** point `DAZZLECMD_CONFIG` at a hand-written `config.json` (UTF-8, with and without a BOM) holding `{"active_kits":["media"]}`; run `dz list`, then `dz kit enable <other>` (triggers a write).
- [ ] **Expected:** reads tolerate the BOM; the subsequent write succeeds and produces clean UTF-8 (no BOM) valid JSON.

## What the automated tests DON'T cover

- **Real atomic write under load** — disk full, permission denied, AV interference mid-write, a crash between temp-write and `os.replace` (filekit's `os.replace` is atomic, but only a real run on a real FS confirms no `.tmp` litter / no partial files).
- **The live editable install** — tests use `tmp_path`; only HV.1/HV.2/Section 4 exercise the user's actual `dz` against a real (temp) config file.
- **Windows codepage / BOM** in real PowerShell `Out-File -Encoding utf8` (writes a BOM) — Section 4.3.
- **Cross-aggregator** (`wtf`, `amdead`) reading their own `properties.json` — only the dz path is smoked here.
- **Performance feel** of config read/write after the filekit indirection (should be identical; eyeball it).
