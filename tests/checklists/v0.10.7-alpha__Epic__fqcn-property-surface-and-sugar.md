## High-Value Verification (~5 minutes)

### HV.1 — the shipping grammar is untouched (PRODUCTION SAFETY)
- [ ] **Steps** (read-only, real install): `dz --version` · `dz list` · `dz info` · `dz kit` · `dz core:safedel --help`
- [ ] **Expected:** all behave exactly as 0.11.16 did; exit 0; no new output on existing surfaces.
- [ ] **Why:** the intercept is the CLI's new FIRST routing rule — if anything shipping regressed, stop here.

### HV.2 — the sugar round-trip
- [ ] **Steps** (temp config): `dz .note "hello world"` → `dz .note` → `dz .note "changed"` → `dz prop delete .note`
- [ ] **Expected:** `added dz.note = 'hello world' (new)` → `hello world` → `updated dz.note = 'changed'` → `deleted dz.note`. The added/updated echo distinguishes create from edit; NO sugar spelling deletes.

### HV.3 — the flagship: channel verbosity with a bare negative
- [ ] **Steps:** `dz :.kit.channels.verbosity -3` → `dz :.kit.channels.verbosity`
- [ ] **Expected:** sets int −3 with NO `--` needed (argparse-parity rule); reads back `-3`.

### HV.4 — dot-anywhere vs all-colons (the taxonomy)
- [ ] **Steps:** `dz :list` (all-colons) · `dz :.author "Dustin"` (forgiveness) · `dz :+kit`
- [ ] **Expected:** `dz :list` behaves exactly like `dz list` (strip-and-dispatch); `:.author` echoes `-> dz.author (canonical)` then adds; `:+kit` errors "reserved (lands with SD-7)", exit 2.

### HV.5 — level is one value everywhere
- [ ] **Steps:** `dz level kit` → `dz use` → `dz .level` → `dz prop get .level` → `dz meta reset` → `dz level`
- [ ] **Expected:** all four reads say `kit`; after reset, `tool`. `dz level bogus` and `dz .level bogus` produce the SAME error, both exit 2.

### HV.6 — the MOVE migration
- [ ] **Steps:** write `{"foreground_level": "kit"}` into the temp config.json by hand, then `dz level`, then view config.json + properties.json.
- [ ] **Expected:** reports `kit`; `foreground_level` is GONE from config.json; `dz.level: "kit"` is in properties.json. Repeat with reset-first (fresh file): `dz meta reset` → `dz level` → `tool` (no resurrection).

## Section 1: value rules
- [ ] 1.1 `dz .note some words` → "ONE token" error, exit 2 (quote multi-word).
- [ ] 1.2 `dz .note --force` → error hinting `--`; `dz .note -- --force` stores `--force`.
- [ ] 1.3 `dz .x vvvv` stores 4 (echo: added/new); `dz .x qqq` on the now-existing key stores -3 (echo: updated) -- the sugar always upserts; the added/updated echo is the create-vs-edit signal.
- [ ] 1.4 `dz -v .note` prints the value AND `-v` produces the verbose channel output (the sugar honors flags).
- [ ] 1.5 `dz -- .note` → intercept disabled → argparse invalid-choice error (POSIX escape).

## Section 2: listing + families
- [ ] 2.1 `dz :.` lists every stored key (properties AND fiber keys together); empty store prints a one-liner.
- [ ] 2.2 Boundary: add `:.kit.channels.verbosity 3` and `:.kitchen.note oops`; `dz :.kit:.` lists kit's family only — kitchen absent.
- [ ] 2.3 `dz :. junk` → "listing takes no value" error.

## Section 3: prop verbs + strictness
- [ ] 3.1 `dz prop set .new 1` → error "use 'prop add'"; `dz prop add .new 1` → ok; `add` again → error "use 'prop set'".
- [ ] 3.2 `dz prop ...` == `dz meta prop ...` (same store key both ways).
- [ ] 3.3 `dz .env-vars:DEBUG 1` then `dz .env-vars:debug 0` → case-collision warning on stderr; both keys stored distinctly.

## Section 4: extended level ladder
- [ ] 4.1 `dz level supra` / `dz level fiber` persist and report; `dz list` still behaves (extended rungs inert at tie-break).
- [ ] 4.2 `dz use -h` no longer shows a fixed `{tool,kit,aggregator}` list (accepted-change surface); invalid rung error mirrors the argparse phrasing, exit 2.
